SshClient, version

Outer World Apps

A basic SSH Client that can be used to access remote computers that are accessible via SSH. Unlike some other Android SSH Clients, this one only requires internet access permission to make SSH connections and write external storage permission for the file transfer functions.

There are five screens:

  • SHELL - contains the currently selected session's contents, consisting of a connect-to single-line text box and a large screen text box. This is the screen you are usually on when communicating with a remote host.
  • FILE TRANSFER - lists the directory you are currently pointed to, either on the remote host or on the local device.
  • TUNNELS - lists the defined tunnels for the current session and provides a method to define new ones, and open, close, delete existing ones.
  • SETTINGS - this has various settings that you can adjust as you like.
  • HELP - this screen.
You can navigate to the desired screen by clicking the Android's menu button then clicking More and then selecting the screen you want.

SHELL Screen

At the top is a single-line text box. To initiate a connection to a host, type in the username@hostnname that you want to communicate with. If it uses a non-standard port number (22), use username@hostname:portnumber format. After entering the information, click the keyboard's Next or Done button to initiate the connection process. If the connection requires a passphrase and/or password, you will be prompted. You may also be asked to confirm the fingerprint of the remote host to prevent man-in-the-middle attacks.

The app will remember what username@hostname[:portnumber]s you have successfully connected to, and provide them as auto-completion text next time you want to connect, so you don't have to retype the long strings on your small keyboard each time.

The color of the text in the host name entry box is significant:

  • BLACK - the app is waiting for the user to enter the information and click Done or Next
  • BLUE - the app is attempting to connect to the remote host
  • GREEN - the connection to the remote host is alive and authenticated and ready for use. You can use the keyboard to send text to the host and the host's replies will appear in the large text area below.
  • RED - the host has disconnected or failed to connect. To reconnect, click on the box and either just click Done/Next to reconnect to same host or change the info first then click Done/Next to connect to a different host.

Once connected, by default, the app will open a shell channel and you can communicate with the host shell using the keyboard. If you want to do file transfers or tunnels instead, you can switch to the file transfer or tunnels screen from the menu at this point and just leave the shell open. Or if you prefer to have the app not open a shell at all, select file transfer or tunnels from the menu BEFORE connecting and the app will switch to that mode as soon as the connection completes without starting shell mode. You can always go back to shell mode later by selecting shell from the menu. Shell mode can also be terminated by typing an exit or similar command to the host so it will close its shell. You can then switch to the file transfer or tunnels mode screen from the menu.

Note: If you are having difficulty getting the soft keyboard to appear for sending text to the host, try clicking on the Android's menu button then selecting shell (again if necessary).

FILE TRANSFER Screen

This screen is used to transfer files between the local device and the current session's remote host. It has these four sections from top to bottom:

  • function buttons - switch between local and remote, initiates functions on selected file
  • selected files - list of files that have been selected
  • navigation buttons - select which directory on either local or remote to look at
  • directory listing - list of the files in the directory selected by the navigation buttons
Notes:
  • Each of the four sections independently scrolls horizontally.
  • You can access a local-files-only version that does not require connection to a remote host by clicking on the Android's menu button, then clicking More then local files only.

Various function buttons may appear on the top row depending on the state of the app. Here are the possibilities, in the order they appear:

  • remote < - navigate directories on the remote host
  • search - search directories for a wildcard pattern
  • make dir - create a sub-directory (must currently be in a writable directory)
  • clear - clear list of selected files (must have something selected)
  • copy - copy selected files to the current directory (all selected files must be readable, the current directory must be writable)
  • move - move selected files to the current directory (all selected files must be readable and in writable directories, and the current directory must be writable)
  • delete - delete the selected files (files must be in the current directory and the current directory must be writable, ie, you can delete files only from the current directory)
  • open - open the one-and-only selected readable file
  • opendir - open the last selected file as a directory (assuming it is a readable directory) OR open the last selected zip file as a read-only directory (assuming it is a readable zip file)
  • > local - navigate directories on the local device
  • The local files only version, accessed via the Android's menu, provides local-1 and local-2 buttons which give two independent views of local files (instead of remote and local buttons which give a view of remote files and a view of local files).

The next row contains navigation buttons that can be used to 'pop up' levels in the directory hierarchy. If a button has black text, indicating a read-only directory, you can only view the directory, you cannot make changes. If a button is red, indicating a read/write directory, you can both view the directory and make changes (add, delete, rename files). Also, eg, on Android, it is possible for you to not have permission to view at an intermediate directory. If this is the case, those intermediate directories will have their names combined with a viewable directory, eg, data/data/com.outerworldapps.sshclient/ all on one button means you can't view either data directory, but you can view the com.outerworldapps.sshclient directory.

Clicking on the last button in the row (which indicates the directory currently being displayed), refreshes the listing by re-reading it. Changes made in one view of a filesystem will not appear in other views of the same filesystem, so use this functionality to display any recent updates.

To descend into a readable directory, click on it in the list (which causes it to highlight) then click the opendir button. The opendir button can also navigate into a readable zip file as if it were a read-only directory. Some patience may be required if the directory or zip file is large.

To copy file(s) (and/or directory tree(s)):

  1. Select a file or directory tree to be copied by clicking on its name in the listing. Its name highlights in the directory listing and the name will also be added to the list of selected files at the top of the screen. If you have permission to read it, the copy button will appear.
  2. To select a range of files, select the first file of the range by clicking on the name, then select the last file of the range by double-clicking on the name.
  3. To deselect a file, click the highlighted name. To deselect all files, click the clear button at the top.
  4. Repeat that step for all files and/or directory trees you want to copy. You can navigate around directories to locate all you want.
  5. Navigate to the directory where you want to copy the file(s). You may copy them to the same directory you are already in if you like. Usually though you want to navigate to a directory on the other system (local vs remote).
  6. Click the copy button. It will prompt you to confirm the copy, allowing you to change the name if you like, then perform the copy. You can cancel a long copy by clicking the Cancel button when the progress box is showing.

Notes:

  • There is no restriction on local vs remote for the copy operation. However, if you do a remote-to-remote copy, the data will transfer from the remote host to the phone then back to the remote host which is very inefficient. You are much better off using the shell to perform such a copy for files of any significant size.
  • When a file is copied to the Android device, Android considers the file as being owned by the SshClient app. To make it accessible to other apps, you should put it somewhere on the SD card storage, which is where the app puts you by default. Similarly, you can only copy files from the Android device that the SshClient app has access to, which is typically just its own files, the files on the SD card storage, plus some read-only system files.
  • While a copy is going (such as for a large file), the app will block you from accessing the same session's shell and tunnels screens, because the TCP connection is being heavily used for the copy and may not allow access to the other streams. You can pause the copy and access the other streams or start another session to the same host to access shell and tunnel while the copy is running.
  • Similarly, while a copy is going, you will be blocked from using the BACK button, in which case you can pause or cancel the copy, then use the BACK button.
  • Pressing the HOME button while a copy is running lets the copy continue in the background, even if Android decides it needs the memory and kills the app, as the service will keep it going.

To move file(s) (or directory tree(s)), perform the same steps as for copying, except use the move button instead of copy. Note that moving a file to the same directory with a different name is a rename operation. Moving a file between the local and remote system involves a copy operation followed by a delete.

The open button will attempt to open the highlighted file if it is readable. If it is a file on the local filesystem, a search is made for an application that can read the file and it is launched. If the file is on the remote host, the file is first copied to a local temporary file directory (called cache on Android), then an application is launched. You may want to either delete or move the file from the cache directory when done with it. Note that the file must also be readable to the application being used to display it, which will be the case for files on the SD card.

TUNNELS Screen

SSH tunnels can forward TCP connections on either the local or remote host over to the other host.

This screen has one button per tunnel defined in the database. If the button has black text, the tunnel is closed and a short click will open or edit it. If the button has red text, the tunnel is open and a short click will close it. In either state, a long click will delete it from the database, closing it first if it is open.

You must already be connected to a remote host for the TUNNELS screen to display. If you attempt to display it before connecting, the opening will be pending until you connect, and then the TUNNELS screen will open. You can also switch to a different screen within the app or press the HOME button at any time and your open tunnels will remain open (until Android decides it needs the memory for something else, that is).

Local => Remote - connections to a port on the local Android device are forwarded to the port on the remote host, encrypted over the SSH connection. If you put 0 for the listening port number, an unused port is picked for you each time the tunnel is opened. Anyway, any connection made on the Android device to the tunnel's listen port are forwarded over the SSH link to the session's remote host, then it is connected to the tunnel's connection hostname:portnumber.

Local <= Remote - connections to a port on the remote host are forwarded to the port on the local Android device, encrypted over the SSH connection. You must specify a specific listening port (1..65535). Any connection made to the tunnel's listen port on the remote host are forwarded over the SSH link back to the Android device then it is connected to the tunnel's connection hostname:portnumber.

SETTINGS Screen

This screen holds customization settings. Use the keyboard or click on the various settings to change them. The changes will not take effect until you click the SAVE button. If you click REVERT, any changes you have made will be discarded and you can make new changes if you like. If you click the Android's menu to change screens (such as shell), your changes will be discarded.

Cursor style - select whether the cursor for the SHELL screen is a thin line, a hollow box, a solid block, or none at all.

Font size - selects the font size used for the SHELL screen and the directory listing part of the FILE TRANSFER screen.

Include hidden files - includes hidden files (files whose name begin with '.') in the directory listing part of the FILE TRANSFER screen.

Max total chars - selects the number of characters that are saved in memory for each SHELL session. Only powers-of-two are permitted here. The larger the number, the farther you will be able to scroll back to see previous host output.

Screen orientation - the orientation for SHELL and FILE TRANSFER screens can be set to:

  • Unlocked - to allow it to change as the device is rotated
  • Portrait - to lock it in Portrait mode regardless of device rotation
  • Landscape - to lock it in Landscape mode regardless of device rotation

Show EOL markers - enables display of a character NL on the SHELL screen that indicates the end-of-line as given by the host.

Text colors - select the colors used for the SHELL screen and the directory listing part of the FILE TRANSFER MODE screen, either black-on-white or white-on-black.

Transfer progress - selects style of file transfer progress bars:

  • Data files only - shows size & time for data files only. Directories are not shown, except briefly between data files.
  • Hierarchy w/out dir sizes - shows size & time for data files, shows file count for directories in the hierarchy.
  • Hierarchy with dir sizes - shows size & time for both data files and directories. Requires pre-scanning the directory tree to gather directory sizes. Directory sizes are the total of all included data files plus some crudely guesstimated per-file overhead.

Wrap long lines - enables wrapping of long lines from the host onto multiple lines of the SHELL screen. Otherwise, long lines can be viewed by horizontally scrolling the SHELL screen.

Note: The screen is locked in Portrait orientation while in the SETTINGS screen.

Menu Buttons

The Android's menu contains these main buttons:

  • Ctrl-... - the next character entered on the keyboard will be shifted to control character, if it is an alphabetic. If a non-alphabetic is entered, the Ctrl-... will be abandoned and the character will be sent normally.
  • Ctrl-C - sends a Control-C character to the host.
  • Freeze - freezes the text in place on the screen (ie, blocks the host from making updates). While frozen, the foreground/background colors are reversed.
  • Paste - sends the contents of a clipboard to the remote host shell.
  • Tab - sends a TAB character to the remote host.
  • More >> - opens up the extended menu (see below).

... and these extended buttons (accessed by clicking the More button in the main menu):

  • clear screen - clears the SHELL screen text window.
  • clipboard <-> file - transfer clipboard to/from file
  • disconnect - disconnects current session, both shell and file transfer.
  • EXIT - after a typical OK/Cancel dialog, immediately closes connection(s) with remote host(s) if any and closes the app.
  • file transfer - opens the FILE TRANSFER screen if connected or queues it for display if not connected. If this is the first time opening it for this session, the remote home directory is scanned and presented.
  • HELP - displays this HELP screen.
  • known hosts - brings up the Known Hosts menu.
  • local files only - opens a FILE TRANSFER screen for navigating local files without being connected to any remote host.
  • local keypairs - brings up the Local Keypair menu.
  • master password - displays menu for setting/changing master password.
  • saved logins - brings up the Saved Logins menu.
  • sessions - selects different sessions.
  • settings - opens the SETTINGS screen.
  • shell - opens the SHELL screen. If connected and the shell is not open, the shell is opened.
  • tunnels - opens the TUNNELS screen.

Known Hosts

Known Hosts are hostnames that you have acknowledged the fingerprint for. If it is the first time you have connected to a particular host with this app on this device, you will be prompted to confirm the fingerprint. From that point on, the app will remember that you have confirmed the fingerprint and will not prompt you again. If you want the app to forget the remembered fingerprint, use this menu to delete it by clicking on it.

Local Keypairs

A Local Keypair is used to identify this app on this device to a remote host. The identity can be used instead of a password so you don't have to store the password on your device and you don't have to type it in each time. Also, some servers may be configured to require keypair authentication and will not accept any password at all.

Once a keypair is generated (using random numbers), the public key must be appended to the server's ~/.ssh/authorized_keys file. If the server is configured to accept a password, you can log in using the password (do not save it) then copy/paste the public key into the server's authorized_keys file. If the server does not allow password logins, you will have to have some other mechanism to accomplish it, such as copy/paste the pubic key into an email that you send to the server's administrator. If the authorized_keys file was properly updated, you will be authenticated using the keypair and will not need to enter a password next time you log in.

When generating a keypair, it asks for the following information:

  • Name to identify - written as a tag on the public key so that at some time in the distant future, when you look at a remote host's ~/.ssh/authorized_keys file, you will know where this particular public key came from, as it will be tagged with this identifier. You can only use a-z, A-Z, 0-9, @, periods, underscores and hyphens in the name, and a maximum length of 50 characters. It is also used to differentiate multiple keypairs on the same device if you create more than one, as each one must have an unique name. The identifier appears at the end of the single line of the public key.
  • Passphrase - used as a password for the keypair file itself. This will safeguard the keypair file in case someone steals your device, they won't be able to access your authorized remote hosts. The same passphrase goes with this keypair file, and since the same keypair file is used for many remote hosts, you only have to remember this one passphrase. Leave both boxes blank if you do not want a passphrase, in which case you should enable a master password.
  • DSA/RSA - keypair encryption format. There are many discussions on the Internet about which is more secure.
  • 1024/2048/4096 - key size. 1024 is the most universally accepted by servers. The bigger the number, the more secure.

You can also load a previously generated keypair that you have saved somewhere. It takes several steps to make this work:

  1. Copy the keypair into the device's clipboard using whatever method you have, possibly from some other app. The text must be formatted to have the public key on the first line and the rest of the text containing the private key, which is several lines long. Typically the public key line begins with something like ssh-rsa and ends with the identifier it was created with. The private key is several lines long and begins with a line like -----BEGIN RSA PRIVATE KEY----- and ends with a line like -----END RSA PRIVATE KEY-----.
  2. Click the Android's menu button then click on local keypairs then select load keypair directly.
  3. Click either the Load from internal clipboard or Load from external clipboard button. You should see a few characters from the beginning of the public key and a few characters from the end of the private key string.
  4. Click the OK button. You should get an acknowledgement message saying the keys were accepted.
You can load the keypair from either the internal clipboard or the external clipboard. For better security, use the internal clipboard wherever possible as it cannot be peeked at by other applications.

Once created on a particular client, a keypair should not be deleted, as they represent an unique key that is used to unlock the hosts that you have added the public key to their ~/.ssh/authorized_keys file.

Caution: While it is perfectly acceptable to copy public keys using any method available, it is considered bad practice to copy private keys around, especially via the external clipboard. The functionality is provided for completeness. You might, at some level of risk, want to back up your private key (along with your public key) by pasting them to a secure host, using the internal clipboard, or use file transfer mode. Use at your own risk!

Master Password

The app stores sensitive information on the device's internal flash memory, namely, the key pair files, the known hosts file and saved logins file which might contain saved passwords. If the device were stolen, anyone could then access hosts authorized by those files, which is undesirable.

To combat this, you should create a master password that must be entered whenever the app is started. It is used to encrypt the files stored on the flash memory so they would be useless to anyone who does not know the master password. You can set and/or change the master password at any time. Changing it causes the files to be re-encrypted with the new master password.

A master password is especially recommended if you have any saved passwords or a keypair without a passphrase. It would be a good idea to at least have a simple PIN-style password (as opposed to none at all) if you do not want a hard-to-enter one. The app places no restriction on what is acceptable, except that it not be blank.

Saved Logins

Saved Logins are the username@hostname[:portnumber] entries that have successfully connected and the corresponding saved password (if you checked the 'Save password' checkbox on the password entry dialog). They also hold the name of the keypair tried, if any. They are displayed as auto-completion suggestions when you make an entry in the user@host[:port] box of the SHELL screen.

If you no longer want one of the entries, use this menu to delete the one you don't want. The menu displays the saved username@hostname[:portnumber] for each entry, and the name of the associated keypair if any, and a 'pw' tag if the password for the entry is saved.

If you want to change or delete a keypair association or a saved password, the only way to do that is to delete the whole saved login entry for that login. Then when you log back in, the entry will be re-created.

Sessions

The app can maintain more than one connection to either the same or different remote hosts. Each connection is referred to as a session. You can create, delete and switch sessions using this menu.

To make a session current, short click on its button. Its contents will be available on the app's shell, file transfer and/or tunnel screens.

To disconnect a session and delete it, long click on its button.

Note: When a session is not current, the app will continue to receive data from the host and update its hidden screen. If you do not want it to update, you can freeze it first before switching away, then unfreeze it when you return back to it.

The Android's status bar will display the number of connected sessions. The sessions will remain active even if you switch to another app, and the status bar line will remind you that there are active sessions.

Freeze Mode

Freeze mode is entered by clicking the Android's menu button then clicking on Freeze. While in freeze mode, data from the host is blocked so the screen won't update. But you can still scroll around. Also, the keyboard will not send characters to the host. Rather, there are special characters that can be used to copy data to a clipboard:

  • b - begin text selection at the cursor
  • c - copy text selection to a clipboard
  • i - move cursor up one line
  • j - move cursor left one character
  • k - move cursor down one line
  • l - move cursor right one line
  • r - reset (discard) selection

To exit freeze mode, click the Freeze button again. The screen colors will go back to normal and the host will be released to update the screen.

Clipboards

Several functions of the app access a clipboard. In general, there are two clipboards these functions have access to. The internal clipboard's contents are only available to the app and can be used to transfer text information from one screen within the app to another. No other app can access the internal clipboard. The external clipboard's contents are available to any app on the device, so you can use it to, for example, transfer text between a connected host and an email message displayed in the web browser.

The app will only access text-oriented clipboard data. It will not properly handle other data such as in image, video and sound files.

The extended menu contains a clipboard <-> file button. Clicking it brings up another menu with these buttons:

  • Internal clipboard -> File - writes the internal clipboard to a file in the current directory opened in the file transfer screen
  • External clipboard -> File - writes the external clipboard to a file in the current directory opened in the file transfer screen
  • File -> Internal clipboard - reads the one-and-only selected file to the internal clipboard
  • File -> External clipboard - reads the one-and-only selected file to the external clipboard
